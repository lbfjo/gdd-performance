## Assistant — iter 2

The spec is solid overall — well-structured, clear scope, and sensible tech choices. Here are the issues I spotted:

### Security

1. **SHA-256 without salt for PINs.** With only 10,000 possible 4-digit PINs, a SHA-256 hash is trivially reversible via a precomputed lookup table (takes milliseconds). The spec should either:
   - Use a salted hash (e.g. per-athlete random salt + SHA-256), or
   - Use bcrypt/scrypt/argon2 (though arguably overkill for a 4-digit PIN — the real issue is that a 4-digit PIN is inherently weak regardless of hashing).
2. **No rate-limiting on PIN attempts.** An attacker (or curious teammate) can brute-force all 10,000 PINs in seconds. Add a lockout or delay after N failed attempts (e.g. 5 failures → 30-second cooldown).
3. **Client-side hashing only.** "Plain-text PINs never leave the client" — but Firestore Security Rules would need to enforce that the `pin` field can't be read directly. The spec doesn't mention Firestore Security Rules at all, which is a significant gap.

### Data Model

4. **No date field on `checkins`.** The "already checked in today" check and the weekly grid both need to query by date, but there's only a `timestamp` (Firestore Timestamp). You'd need to either:
   - Add a `date: string` field (e.g. `"2026-06-03"`) for easy equality queries, or
   - Do range queries on `timestamp` — which works but is more verbose and timezone-sensitive.
5. **No compound index mentioned.** Queries like "all check-ins for athlete X in week Y" or "all check-ins for date D" will need Firestore composite indexes. Worth noting.
6. **`athleteName` denormalized but no sync strategy.** If an athlete's name changes, stale names persist in all historical check-ins. Fine for a POC, but worth calling out.

### Firestore Free Tier

7. **Read limits may be tighter than expected.** The weekly grid loads all athletes × 7 days of check-ins per coach visit. With ~100 athletes and say 5 check-ins/athlete/week, that's ~500 check-in docs + ~100 athlete docs per dashboard load. 10 coaches refreshing a few times a day eats into the 50K reads/day budget. It'll probably be fine, but the spec's claim that limits are "more than sufficient" deserves a back-of-napkin calculation.

### UX / Functional Gaps

8. **Timezone not specified.** "Already checked in today" and week boundaries (Mon–Sun) depend on timezone. The spec should state which timezone governs (presumably Portugal/Lisbon).
9. **Week definition not specified.** "Current week" — ISO week (Mon–Sun)? Or something else? Matters for the grid and leaderboard.
10. **Auto-reset after 5 seconds with no cancel.** If an athlete checks in and wants to verify their weekly count, 5 seconds is short. Consider a "Done" button that also triggers reset.

### Minor / Cosmetic

11. **PWA offline message says "No internet connection" but the check-in list itself is fetched from Firestore.** Without caching the athlete list, the app is completely non-functional offline — not just "can't check in" but "blank screen." The spec should clarify that even the athlete list requires connectivity.
12. **No mention of data retention.** Over multiple off-seasons, the `checkins` collection grows unbounded. Low priority for a POC, but worth a one-liner.

None of these are showstoppers for a POC, but items 1–3 (security) and 8–9 (timezone/week definition) should be addressed before build.
